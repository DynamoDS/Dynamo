# Quality Checklist

Run these checks after making code changes to avoid masking real problems.

## Anti-patterns to catch

| Pattern | Why it's a problem |
|---|---|
| `[Ignore]` or `[Explicit]` on tests without a comment | Silently skips coverage |
| `catch { }` or `catch (Exception) { }` with no logging | Swallows errors, hides bugs |
| `#pragma warning disable` without a justification comment | Hides real issues |
| `Assert.AreEqual` weakened to `Assert.IsNotNull` | Reduces test signal |
| Hardcoded magic numbers | Should be named constants |

These are signs of reward hacking — making CI pass without fixing the underlying issue.

## Security rules (treated as errors)

- CA2327, CA2328, CA2329, CA2330 — do not suppress without documented justification.
- Never commit secrets or credentials.
- No new network connections without documentation and no-network mode testing.
- No data collection without user consent checks.
- User-facing strings must go in `.resx` files for localization.

## File constraints

- No files > 50 MB.
- Don't edit auto-generated code: files matching `/api/*.cs`, `*.g.cs`, or marked `// <auto-generated>`.

## PR / commit checklist

- PR title includes Jira ticket: `DYN-1234 concise summary`.
- All sections of `.github/PULL_REQUEST_TEMPLATE.md` filled in.
- Release Notes field: mandatory — use `N/A` if not user-facing (minimum 6 words otherwise).
- No breaking API changes without a filed issue and Semantic Versioning review.
